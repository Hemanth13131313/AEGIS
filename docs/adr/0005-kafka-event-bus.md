# Use Kafka as central event bus for async analysis

## Status
Accepted

## Context
AEGIS requires processing high volumes of events generated by AI interactions. The gateway needs to enforce policies synchronously (data plane), but complex ML models for prompt injection, jailbreak detection, and anomaly detection (e.g., RAG poisoning) are too slow (hundreds of ms) to run in the critical enforcement path (budget < 10ms). 

To ensure asynchronous scanners and monitors receive events reliably without impacting proxy performance, a durable message bus is needed.

## Decision
We will use **Confluent Kafka** (running in KRaft mode) as the central event bus for the async analysis plane.
- The `session_id` will be used as the Kafka partition key to ensure ordered processing per session.
- The `aegis.events.raw` topic will store high-throughput raw events (retained for 7 days).
- The `aegis.events.detections` topic will store security events emitted by scanners (retained for 30 days).

## Consequences
- **Positive:** Complete decoupling of the synchronous enforcement path from async ML analysis. Kafka provides high throughput, durability, and allows multiple consumer groups (Scanner, RAG Monitor, ClickHouse writer, etc.) to process the same events at their own pace.
- **Negative:** Increased infrastructure complexity and memory footprint (Kafka JVM). Requires at-least-once consumer semantics and idempotent downstream systems.
